## 概述

本文件详细梳理基于语义化版本号（Semantic Versioning）的三种版本开发流程，分别对应：

1. **PATCH 版本**（vX.X.1）：Bug 修复
2. **MINOR 版本**（vX.1.0）：新增功能，向后兼容
3. **MAJOR 版本**（v1.0.0）：不兼容的 API 变更

```

  │     │     │

```

## PATCH 版本开发流程

### 适用场景

- 修复生产环境发现的 Bug
- 修复回归问题
- 性能优化（小范围）
- 文档修复
- 类型定义修复

### 推荐使用 Skill Chain

### 详细步骤

#### 步骤 1: Bug 确认和分析

**技能链：** `fix-test-failure` / `fix-type-error` / `fix-build-error`

1. **问题复现**
   - 创建最小复现案例
   - 确认问题在当前版本存在

2. **根因分析**
   - 使用 `debug-code` 技能
   - 定位问题代码
   - 理解为什么会出现问题

3. **影响评估**
   - 确定影响范围
   - 评估严重程度
   - 决定是否需要紧急发布

#### 步骤 2: 创建修复分支

**技能：** `create-branch`

```bash
# 创建修复分支
git checkout -b fix/xxx-issue-id
```

**分支命名规范：**

- `fix/{问题描述}-{issue-id}`
- 示例：`fix/type-error-compiler-123`

#### 步骤 3: 修复实现

**核心原则：最小化变更**

1. **修复代码**
   - 只修改与问题相关的代码
   - 不要引入无关变更
   - 保持 API 兼容性

2. **添加测试**
   - 添加回归测试，防止问题再次出现
   - 确保测试覆盖修复场景

3. **代码规范**
   - 使用 `code-style` 技能
   - 遵循项目代码风格

#### 步骤 4: 本地验证

**技能链：** `type-check` → `lint-code` → `run-tests`

```bash
# 运行完整验证
pnpm type-check
pnpm lint:check
pnpm test
```

#### 步骤 5: 代码审查

1. **提交变更**

   ```bash
   git add .
   git commit -m "fix: 描述修复内容"
   git push origin fix/xxx-issue-id
   ```

2. **创建 Pull Request**
   - 标题格式：`fix: {简短描述}`
   - 描述问题、根因、解决方案
   - 关联 issue

3. **审查通过**
   - 至少一人审查通过
   - 所有 CI 检查通过

#### 步骤 6: 合并到 develop

```bash
git checkout develop
git pull origin develop
git merge --no-ff fix/xxx-issue-id
git branch -d fix/xxx-issue-id
git push origin develop
```

#### 步骤 7: 发布 PATCH 版本（如需要）

参考[通用发布流程](#通用发布流程)

### 详细步骤

#### 步骤 1: 需求规划

**技能：** `create-spec`（复杂功能） / `create-plan`（中等功能）

1. **需求分析**
   - 收集用户反馈
   - 确定功能目标
   - 定义成功指标

2. **技术设计**
   - API 设计
   - 架构方案
   - 兼容性评估

3. **更新 Roadmap**
   - 更新 `docs/contribute/roadmap/current.md`
   - 列出功能清单
   - 分配优先级

#### 步骤 2: 创建功能分支

**技能：** `create-branch`

```bash
# 创建功能分支
git checkout -b feature/xxx
```

**分支命名规范：**

- `feature/{功能描述}`
- 示例：`feature/compiler-cache-optimization`

#### 步骤 3: 功能开发

**技能链：** `version-upgrade`

**开发原则：**

- 模块化开发，逐个实现
- 每个模块有完整测试
- 保持向后兼容

**每个模块的子流程：**

1. 实现功能代码
2. 编写单元测试
3. 运行 `pnpm type-check`
4. 运行 `pnpm lint:check`
5. 运行 `pnpm test`
6. 提交代码

**提交信息格式：**

```
feat({scope}): 描述新增功能

详细描述...
```

#### 步骤 4: 集成测试

1. **完整测试**

   ```bash
   pnpm test
   ```

2. **性能测试**（如适用）

   ```bash
   pnpm bench
   ```

3. **手动测试**
   - 使用 playground 测试完整功能
   - 验证用户场景

#### 步骤 5: 文档更新

**技能：** `documentation`

需要更新的文档：

- API 文档（`docs/api/`）
- 用户指南（`docs/guide/`）
- 示例代码（`examples/`）
- CHANGELOG（待发布时生成）

#### 步骤 6: 代码审查与合并

1. **提交 PR**
   - 标题：`feat: {功能描述}`
   - 描述功能、使用方法、兼容性

2. **代码审查**
   - 至少两人审查
   - 确保代码质量

3. **合并**
   ```bash
   git checkout develop
   git pull origin develop
   git merge --no-ff feature/xxx
   git branch -d feature/xxx
   git push origin develop
   ```

#### 步骤 7: 发布 MINOR 版本

参考[通用发布流程](#通用发布流程)

## MAJOR 版本开发流程

### 适用场景

- 架构重构
- 不兼容的 API 变更
- 核心功能重写
- 大规模性能优化
- 移除废弃 API

### 推荐使用 Skill Chain

使用 [major-version](../../.trae/skills/skill-chains/SKILL.md#major-version---major-版本开发) 技能链

┌─────────────────────────────────────────────────────────────────┐

### 详细步骤

#### 步骤 1: 战略规划

**技能：** `create-spec`

1. **愿景定义**
   - 明确 MAJOR 版本的目标
   - 定义成功标准
   - 确定时间表

2. **架构设计**
   - 新架构方案
   - 迁移策略
   - 兼容性方案

3. **风险评估**
   - 技术风险
   - 生态影响
   - 应对措施

#### 步骤 2: 创建 MAJOR 分支

```bash
# 从 develop 创建 next 分支
git checkout develop
git pull origin develop
git checkout -b v2.0.0-dev
# 或
git checkout -b next
```

#### 步骤 3: 分阶段开发

**阶段 1：核心架构重构**

- 建立新的架构基础
- 实现核心功能
- 确保架构稳定

**阶段 2：功能迁移与实现**

- 逐个迁移现有功能
- 实现新功能
- 保持测试覆盖

**阶段 3：兼容性层（可选）**

- 提供迁移辅助工具
- 兼容性适配器
- 逐步废弃旧 API

**每个阶段的流程：**

1. 创建子分支：`feature/v2-xxx`
2. 开发功能
3. 测试验证
4. 合并到 MAJOR 分支

#### 步骤 4: Alpha 版本

**目标：** 内部测试，功能冻结

1. **发布 Alpha 版本**

   ```bash
   # 版本号：vX.0.0-alpha.1
   pnpm run versions:update -- preid alpha
   ```

2. **内部测试**
   - 完整测试套件
   - 性能基准测试
   - 文档完整性检查

3. **Bug 修复**
   - 修复发现的问题
   - 更新文档

#### 步骤 5: Beta 版本

**目标：** 公开测试，收集反馈

1. **发布 Beta 版本**

   ```bash
   # 版本号：vX.0.0-beta.1
   pnpm run versions:update -- preid beta
   ```

2. **公开测试**
   - 发布到 npm（带 beta tag）
   - 社区测试
   - 收集反馈

3. **调整优化**
   - 根据反馈调整
   - 完善文档
   - 性能优化

#### 步骤 6: RC 版本

**目标：** 发布候选，代码冻结

1. **发布 RC 版本**

   ```bash
   # 版本号：vX.0.0-rc.1
   pnpm run versions:update -- preid rc
   ```

2. **最终测试**
   - 完整回归测试
   - 性能基准对比
   - 安全审计

3. **修复阻断性问题**
   - 只修复阻断性 Bug
   - 不引入新功能

#### 步骤 7: 正式发布

参考[通用发布流程](#通用发布流程)

## 通用发布流程

无论哪种版本类型，发布阶段都遵循相同的流程。

### 推荐使用 Skill Chain

使用 [version-release](../../.trae/skills/skill-chains/SKILL.md#version-release---版本发布阶段发布流程) 技能链

### 技能链

使用 `version-release` 技能链

### 发布前检查清单

- [ ] 所有测试通过：`pnpm test`
- [ ] 类型检查通过：`pnpm type-check`
- [ ] Lint 检查通过：`pnpm lint:check`
- [ ] 构建成功：`pnpm build`
- [ ] 文档已更新
- [ ] CHANGELOG 已生成
- [ ] Roadmap 已准备归档

### 发布步骤

#### 步骤 1: 最终验证

```bash
# 完整质量检查
pnpm type-check
pnpm lint:check
pnpm test
pnpm build
```

#### 步骤 2: 版本管理

**技能：** `manage-version`

```bash
# 同步所有包版本
pnpm run sync-versions

# 或手动更新
pnpm run versions:update -- {major|minor|patch}
```

#### 步骤 3: 生成 CHANGELOG

**技能：** `generate-changelog`

```bash
# 使用脚本生成 CHANGELOG
# 或手动更新 CHANGELOG.md
```

#### 步骤 4: 提交版本变更

```bash
git add .
git commit -m "chore: release vX.X.X"
git tag vX.X.X
git push origin develop
git push origin vX.X.X
```

#### 步骤 5: 智能构建

**技能：** `smart-build`

```bash
# 按依赖顺序构建
pnpm tsx scripts/smart-build.ts
```

#### 步骤 6: 发布到 npm

**技能：** `npm-publish`

```bash
# 1. 准备发布
pnpm run publish:prepare

# 2. 发布所有包
pnpm run publish:all

# 3. 恢复 workspace
pnpm run publish:restore
```

#### 步骤 7: Roadmap 归档

**技能：** `roadmap-management`

1. 将 `current.md` 移动到 `archive/vX.X.X.md`
2. 创建新的 `current.md` 用于下一版本
3. 更新 `index.md`

#### 步骤 8: 任务复盘

- 回顾整个版本开发过程
- 总结经验教训
- 识别改进点

## 常见问题

### Q: 什么时候应该发布 PATCH 版本？

A: 当有严重的 Bug 影响用户使用时，应该尽快发布 PATCH 版本。对于非紧急的 Bug，可以积累多个修复后一起发布。

### Q: MINOR 版本可以包含 Bug 修复吗？

A: 可以。MINOR 版本通常包含新功能和 Bug 修复的组合。但 Bug 修复也可以单独通过 PATCH 版本发布。

### Q: MAJOR 版本开发周期一般多长？

A: 根据项目复杂度，MAJOR 版本开发周期可能从几周到几个月不等。建议分阶段发布 Alpha/Beta/RC 版本，逐步稳定。

### Q: 如何保持向后兼容性？

A:

- PATCH/MINOR 版本必须保持向后兼容
- 废弃 API 时先添加 `@deprecated` 注释
- 提供迁移工具和文档
- MAJOR 版本可以考虑提供兼容性层

### Q: 发布后发现严重 Bug 怎么办？

A:

1. 立即创建 PATCH 分支修复
2. 尽快发布 PATCH 版本
3. 更新文档说明问题和解决方案
