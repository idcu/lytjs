# LytJS AGENTS 架构优化指南

## 现状分析

### 当前架构概览

项目目前采用 8 层级的 Skill 架构：

```
原子级 (1-atomic) → 分子级 (2-molecular) → 无机物级 (3-inorganic) → 有机物级 (4-organic)
→ 细胞级 (5-cellular) → 组织级 (6-tissue) → 器官级 (7-organ) → 生物体级 (8-organism)
```

**当前状态**：

- 共 36+ 个 Skill 文件
- 包含多个索引和架构文档
- 层级分明但复杂度较高

### 发现的问题

#### 1. 层级过多导致复杂度过高 ⚠️

**问题**：

- 8 个层级对于实际使用来说过于复杂
- 很多层级之间的边界不够清晰
- 增加了理解和维护的成本

**示例**：

- 器官级和组织级的功能有重叠
- 有些 Skill 可以直接从原子级跳到有机物级

#### 2. 内容重复和冗余 ⚠️⚠️

**问题**：

- 多个 Skill 包含相似的 Checklist
- 同样的命令参考出现在多个地方
- quick-reference 同时存在于根目录和 1-atomic 目录

**影响**：

- 浪费 Token
- 维护困难，需要同步更新多处
- 容易造成信息不一致

#### 3. 部分 Skill 过大，职责不单一 ⚠️

**问题**：

- 如 `dev-guide`、`full-agents-experience` 等包含太多内容
- 违背了原子级"单一职责"的原则
- 难以复用和维护

#### 4. 路径使用不统一 ⚠️

**问题**：

- 有些使用绝对路径 `file:///e:/trae/lytjs/...`
- 有些使用相对路径 `.trae/skills/...`
- 有些使用相对链接 `../../...`

**影响**：

- 在不同环境下可能失效
- 增加了维护成本

#### 5. 索引和导航体验可以改进 ⚠️

**问题**：

- 虽然有 INDEX.md，但导航不够直观
- 缺少按使用频率的分类
- 缺少快速入口

---

## 优化建议

### 建议 1：简化层级架构（推荐）✅

**目标**：从 8 层简化为 4 层，更清晰易用

**新架构**：

```
1. 基础层 (base)  - 原子级 + 分子级的合并
2. 功能层 (feature) - 无机物级 + 有机物级的合并
3. 工作流层 (workflow) - 细胞级 + 组织级的合并
4. 核心层 (core) - 器官级 + 生物体级的合并
```

**优势**：

- 层级减少 50%
- 更符合实际使用场景
- 更容易理解和导航

---

### 建议 2：创建统一的索引和导航 ✅

**建议创建**：

1. **主索引** (`INDEX.md`) - 简化版，聚焦常用
2. **快速参考卡片** (`QUICK_REFERENCE_CARD.md`) - 一页纸式参考
3. **任务决策树** (`TASK_DECISION_TREE.md`) - 可视化决策路径

**示例快速参考卡片**：

```markdown
# 🚀 LytJS 快速参考

## 我要做什么？

| 任务     | 直接调用                                                           |
| -------- | ------------------------------------------------------------------ |
| 新功能   | [workflow:feature-dev](.trae/skills/workflow/feature-dev/SKILL.md) |
| 修复 Bug | [workflow:bug-fix](.trae/skills/workflow/bug-fix/SKILL.md)         |
| 写测试   | [feature:testing](.trae/skills/feature/testing/SKILL.md)           |
| 查命令   | [base:commands](.trae/skills/base/commands/SKILL.md)               |
```

---

### 建议 3：合并冗余内容 ✅✅

**具体行动**：

1. 合并两个 `quick-reference` - 保留原子级的
2. 创建统一的 `commands` Skill - 所有命令集中管理
3. 创建统一的 `checklists` Skill - 所有检查清单集中管理
4. 从高级 Skill 中移除重复内容，改为引用基础 Skill

**优势**：

- 节省约 30-40% 的 Token
- 维护更容易
- 信息更一致

---

### 建议 4：重构大 Skill 为小组合 ✅

**示例**：将 `dev-guide` 拆分为：

```
dev-guide/
├── index.md (主入口，简洁)
├── coding.md (编码规范)
├── testing.md (测试指南)
└── git.md (Git 工作流)
```

**优势**：

- 每个文件更小，加载更快
- 更容易复用其中的某一部分
- 更符合单一职责原则

---

### 建议 5：统一路径规范 ✅

**规范**：

- 统一使用**相对路径**，相对于项目根目录
- 格式：`.trae/skills/[层级]/[skill-name]/SKILL.md`

**示例**：

```markdown
✅ 正确：[quick-reference](.trae/skills/base/quick-reference/SKILL.md)
❌ 避免：[quick-reference](file:///e:/trae/lytjs/.trae/skills/...)
❌ 避免：[quick-reference](../../1-atomic/quick-reference/SKILL.md)
```

---

### 建议 6：优化 AGENTS.md ✅✅✅

**当前 AGENTS.md 的优点**：

- 结构清晰
- 任务类型映射明确
- 核心规则突出

**可以改进的地方**：

1. 添加更明确的优先级指示
2. 简化 Checklist，避免过长
3. 添加"快速开始"一节
4. 优化格式，更易扫描
